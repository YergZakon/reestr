# Миграции схемы БД

## Журнал применения
Источник истины по применённым миграциям — таблица **`schema_migrations`** в продуктивной БД
(`version` PK, `filename`, `checksum` md5, `applied_at`, `note`). Заведена и заполнена
ретроспективно 001–024 (backfill 2026-07-02).

## Правило (обязательное)
1. Любое изменение схемы — **новым файлом** `NNN_краткое_имя.sql` (не правкой существующего/не inline-DDL в коде).
2. Файл идемпотентен: `CREATE TABLE IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`.
3. После применения — запись в `schema_migrations` (version, filename, md5).
4. Не менять уже применённый файл (изменится checksum → рассинхрон). Исправление — только новой миграцией.

## Применение
`python scripts/apply_migrations.py` (веб-репо) — прогоняет недостающие по журналу.
Ретроспективный backfill выполнен разово; далее — только через журнал.

## Технический долг (см. docs/architecture/09, Н4)
DDL сейчас в трёх местах: эти миграции, inline `initDB()` в `src/lib/db.ts`, и `CREATE TABLE`
в отдельных Python-скриптах конвейера (`legal_monitoring`, `ersop_*`, сиды). Цель — свести
всё порождение схемы к каталогу миграций + журналу.
